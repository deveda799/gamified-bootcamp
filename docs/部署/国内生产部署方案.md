# 国内生产部署方案

## 1. 目标与边界

本方案面向中国大陆用户，正式生产环境统一部署在国内云平台。

目标技术栈：

- Node.js
- Next.js
- 国内 Linux 服务器
- Nginx
- PostgreSQL（国内）
- Docker（可选）

明确不作为正式生产方案：

- Vercel
- Supabase Cloud
- 海外数据库或跨境依赖
- Kubernetes（当前阶段不需要）

本方案不要求修改业务代码、页面或 Next.js 标准目录结构。

## 2. 当前项目兼容性

当前项目保持标准 Next.js App Router 结构：

```text
app/
components/
lib/
public/
package.json
next.config.ts
pnpm-lock.yaml
```

已有标准运行脚本：

```bash
pnpm build
pnpm start
```

因此可以直接部署为常驻 Node.js 服务，由 Nginx 反向代理。

当前 H5 MVP 使用本地 mock 数据和浏览器 `localStorage`，不依赖数据库即可部署。PostgreSQL 是未来接入真实业务数据时的生产数据库目标，不是当前 H5 上线的前置条件。

## 3. 推荐生产架构

```text
中国大陆用户
    │
    ▼
国内 DNS / 已备案域名
    │
    ▼
Nginx（HTTPS、反向代理、静态缓存）
    │
    ▼
Next.js Node.js 服务（127.0.0.1:3000）
    │
    ▼
国内 PostgreSQL（未来真实数据阶段）
```

基本原则：

- Nginx 只开放 `80/443`。
- Next.js 的 `3000` 端口只监听本机或服务器内网。
- PostgreSQL 优先使用云厂商托管数据库，并通过同地域内网连接。
- 应用、数据库和对象存储应选择同一地域，避免跨地域延迟。

## 4. 支持的国内云平台

| 云平台 | 应用服务器 | PostgreSQL |
| --- | --- | --- |
| 阿里云 | 轻量应用服务器或 ECS | RDS PostgreSQL |
| 腾讯云 | 轻量应用服务器或 CVM | TencentDB for PostgreSQL |
| 华为云 | Flexus 云服务器或 ECS | RDS for PostgreSQL |

MVP 初期建议：

- 应用服务器：2 核 4 GB 起步。
- 系统：Ubuntu 22.04 LTS 或 24.04 LTS。
- 系统盘：40 GB 起步。
- 带宽：根据实际访问量选择，初期可从 3–5 Mbps 起步。
- 地域：选择主要用户集中区域，并确保应用与数据库同地域。

中国大陆域名正式提供 Web 服务前，应完成域名实名认证、ICP备案、公安备案和 HTTPS 配置。

## 5. 方案 A：Node.js + systemd（推荐）

该方案改动最少，适合当前 MVP。

### 5.1 安装运行环境

在服务器安装：

- Node.js 22 LTS
- Corepack
- pnpm 11
- Git
- Nginx

示例：

```bash
node --version
corepack enable
pnpm --version
nginx -v
```

### 5.2 部署目录

建议统一使用：

```text
/opt/gamified-bootcamp/current
```

代码可通过私有 Git 仓库 Deploy Key、CI 发布包或人工上传进入该目录。任何真实密钥不得写入仓库。

### 5.3 安装与构建

```bash
cd /opt/gamified-bootcamp/current
pnpm install --frozen-lockfile
pnpm test
pnpm build
```

构建通过后再重启生产服务。

### 5.4 systemd 服务

创建：

```text
/etc/systemd/system/gamified-bootcamp.service
```

示例配置：

```ini
[Unit]
Description=Gamified Bootcamp Next.js
After=network.target

[Service]
Type=simple
User=www-data
Group=www-data
WorkingDirectory=/opt/gamified-bootcamp/current
Environment=NODE_ENV=production
ExecStart=/usr/bin/pnpm start --hostname 127.0.0.1 --port 3000
Restart=always
RestartSec=5
NoNewPrivileges=true

[Install]
WantedBy=multi-user.target
```

首次启用：

```bash
sudo systemctl daemon-reload
sudo systemctl enable gamified-bootcamp
sudo systemctl start gamified-bootcamp
sudo systemctl status gamified-bootcamp
```

查看日志：

```bash
sudo journalctl -u gamified-bootcamp -f
```

`pnpm` 的实际路径以服务器执行 `which pnpm` 的结果为准；若路径不同，需要同步修改 `ExecStart`。

## 6. Nginx 配置

创建：

```text
/etc/nginx/sites-available/gamified-bootcamp
```

示例：

```nginx
server {
    listen 80;
    server_name camp.example.com;

    client_max_body_size 20m;

    location /_next/static/ {
        proxy_pass http://127.0.0.1:3000;
        proxy_cache_valid 200 365d;
        add_header Cache-Control "public, max-age=31536000, immutable";
    }

    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_read_timeout 60s;
    }
}
```

启用并检查：

```bash
sudo ln -s /etc/nginx/sites-available/gamified-bootcamp /etc/nginx/sites-enabled/gamified-bootcamp
sudo nginx -t
sudo systemctl reload nginx
```

正式上线必须配置 HTTPS。证书可以使用云厂商 SSL 证书服务或受信任的自动化证书工具。配置完成后，应将 HTTP 请求统一重定向到 HTTPS。

## 7. 方案 B：Docker（可选）

Docker 适合需要统一构建环境、快速回滚或后续接入 CI/CD 的场景。

推荐容器结构：

```text
Nginx（宿主机或独立容器）
    │
    ▼
Next.js 容器：3000
    │
    ▼
云厂商托管 PostgreSQL
```

原则：

- 使用 Node.js 22 官方基础镜像。
- 在镜像构建阶段执行 `pnpm install --frozen-lockfile` 和 `pnpm build`。
- 容器启动命令使用 `pnpm start --hostname 0.0.0.0 --port 3000`。
- 不把 PostgreSQL 与应用放在同一个生产容器中。
- 不把密钥写进 Dockerfile 或镜像。
- 使用环境变量文件或云平台密钥管理服务注入配置。
- 镜像使用明确版本号，禁止只使用不可追踪的 `latest`。

当前项目无需修改 `next.config.ts` 即可按标准 Node.js 模式运行。若未来需要 Next.js `standalone` 镜像优化，应作为单独部署优化处理，不与业务开发混在一起。

## 8. 国内 PostgreSQL 规划

当前 mock H5 不需要 PostgreSQL。未来接入真实数据时，建议使用云厂商托管 PostgreSQL：

- PostgreSQL 15 或 16。
- 数据库与应用服务器处于同地域、同 VPC。
- 禁止向公网开放 `5432`。
- 单独创建应用账号，禁止使用数据库超级管理员运行应用。
- 开启自动备份、时间点恢复和慢查询日志。
- 正式环境与测试环境使用不同实例或不同数据库账号。
- 连接字符串只通过服务端环境变量注入。

环境变量示例只能使用假值：

```env
DATABASE_URL=postgresql://app_user:<REDACTED>@postgres.internal:5432/gamified_bootcamp
```

未来业务代码接入数据库时，应保留现有 Next.js 页面路由和前端交互，只替换数据访问实现。

## 9. 发布流程

推荐采用以下固定流程：

```text
拉取已审核代码
→ pnpm install --frozen-lockfile
→ pnpm test
→ pnpm build
→ 重启 Next.js
→ 检查 Nginx
→ 执行页面 Smoke Test
```

更新示例：

```bash
cd /opt/gamified-bootcamp/current
git pull --ff-only
pnpm install --frozen-lockfile
pnpm test
pnpm build
sudo systemctl restart gamified-bootcamp
sudo systemctl reload nginx
```

禁止在正在运行的开发服务目录中同时执行生产构建。生产服务器只运行 `pnpm start`，不运行 `pnpm dev`。

## 10. 回滚方案

每次上线前保留：

- 上一个 Git commit 或发布标签。
- 上一个可运行构建目录或 Docker 镜像。
- 数据库变更前备份。

非 Docker 回滚：

```bash
git checkout <LAST_GOOD_COMMIT>
pnpm install --frozen-lockfile
pnpm build
sudo systemctl restart gamified-bootcamp
```

Docker 回滚：将运行镜像切回上一个明确版本号，然后重新启动容器。

数据库迁移必须提供向前修复方案；不要在未备份时直接执行破坏性回滚。

## 11. 安全与运维要求

- 仅开放 `22`、`80`、`443`，SSH 端口限制来源 IP。
- 禁止公网直接访问 Next.js `3000` 和 PostgreSQL `5432`。
- 使用非 root 用户运行 Node.js 服务。
- 所有敏感变量放在服务器环境或云密钥管理服务中。
- 日志不得输出密码、Token、Cookie 或数据库连接串。
- 开启服务器监控、磁盘告警、进程存活检查和 HTTPS 到期提醒。
- PostgreSQL 定期执行备份恢复演练。
- 操作系统、Node.js 和依赖按固定维护窗口升级。

## 12. 上线验收

部署完成后至少验证：

```text
/
/app/home
/app/course
/app/course/day-4
/app/submit/day-4
/app/growth
```

验收标准：

- 域名可以通过 HTTPS 访问。
- HTTP 自动跳转 HTTPS。
- 首页和 5 个 MVP 页面正常打开。
- 页面刷新后无 404。
- 静态资源加载成功。
- 移动端无横向滚动。
- Nginx 无持续 `502/504`。
- Next.js 服务重启后自动恢复。
- 当前 mock H5 不请求 Supabase 或海外 API。

## 13. 最终部署决策

当前正式生产方案确定为：

```text
国内 Linux 服务器
+ Node.js 22 LTS
+ Next.js
+ pnpm
+ systemd
+ Nginx
+ HTTPS
+ 国内 PostgreSQL（真实数据阶段）
```

云平台可选：

- 阿里云轻量应用服务器或 ECS
- 腾讯云轻量应用服务器或 CVM
- 华为云 Flexus 或 ECS

Docker 保留为可选部署方式，不作为当前 MVP 上线的必选项。
